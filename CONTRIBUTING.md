# Contributing

感谢你为本项目做贡献！

## 分支策略

- `main`：稳定分支，仅接受经过评审的合并。
- `feature/*`：新功能开发分支。
- `fix/*`：问题修复分支。
- `docs/*`：文档更新分支。

建议从最新 `main` 创建分支，并保持“一次变更一个主题”。

## 提交问题前
- 先搜索是否已有相同 Issue。
- 提供复现步骤、预期结果、实际结果。
- 如涉及界面问题，请附截图或录屏。

## 提交 Pull Request
- 每次 PR 聚焦一个主题，避免混入无关改动。
- 代码需保持清晰，命名语义化。
- 新功能请补充必要文档；涉及行为变更请更新 README。
- 提交前请确保本地可运行、关键流程可验证。
- 至少 1 位维护者通过 Review 后再合并。
- 如涉及 UI 改动，请附截图或录屏。
- 如涉及数据库结构改动，请附迁移说明与回滚方案。

## PR 检查清单

- [ ] 已自测核心流程，结果符合预期。
- [ ] 未引入无关改动。
- [ ] 文档已同步更新（如适用）。
- [ ] 风险点与回滚方式已说明（如适用）。

## Commit 规范
- 推荐使用 `type(scope): subject` 格式。
- 常用 type：`feat`、`fix`、`docs`、`refactor`、`build`、`chore`。

## 代码风格
- 避免重复逻辑，公共能力请抽取复用。
- 变更应尽量小而明确，便于评审和回滚。

## 安全与配置

- 不要提交真实密码、Token、Webhook、私钥等敏感信息。
- 本地调试配置请使用环境变量或本地私有配置文件。
- 发现安全问题请不要公开披露，请按 `SECURITY.md` 指引上报。

## 行为准则
请先阅读并遵守 `CODE_OF_CONDUCT.md`。
